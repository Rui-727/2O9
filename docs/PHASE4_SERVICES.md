# Phase 4: Services

2O9 Phase 4 declares systemd services in `2O9.nix` under `services`. Each
entry says whether the service is enabled, what it depends on, and (when
2O9 should generate the unit file) how to start it. `209 apply`
topologically sorts services by their `requires` dependencies and
activates them in that order.

## Config schema

```nix
{
  services = {
    sshd = {
      enable = true;
      requires = [ "network" ];
      after = [ "network.target" ];
      before = [ "multi-user.target" ];
      restartOnChange = true;
    };
    nginx = {
      enable = true;
      requires = [ "network" "sshd" ];
      after = [ "network-online.target" ];
    };
    "custom-app" = {
      enable = true;
      requires = [ "nginx" ];
      type = "simple";
      execStart = "/nix/store/<hash>-custom-app-1.0/bin/app";
      environment = {
        PORT = "8080";
        DB_HOST = "localhost";
      };
    };
  };
}
```

Services without `execStart` are treated as pre-existing system services.
2O9 just enables and starts them; it does not generate a unit file. The
`sshd` and `nginx` entries above fall into this category. The
`custom-app` entry sets `execStart`, so 2O9 writes a unit file at
`/etc/systemd/system/custom-app.service`.

## Fields

| Field | Type | Default | Notes |
|---|---|---|---|
| `enable` | bool | false | Whether to enable and start the service |
| `requires` | list of string | `[]` | Other service names this depends on. Used for the topological sort and the unit file's `Requires=` line. |
| `after` | list of string | `[]` | Freeform systemd unit names for `After=`. Not validated against the manifest; pass targets like `network.target`. |
| `before` | list of string | `[]` | Freeform systemd unit names for `Before=`. |
| `restartOnChange` | bool | false | Restart on `209 apply` if the service is running and the generated unit file changed. |
| `type` | string | `"simple"` | systemd `Service.Type=`. Used only when `execStart` is set. |
| `execStart` | string | (none) | Command to run. When set, 2O9 generates the unit file. |
| `environment` | attrset of string | `{}` | `Environment=` entries in the unit file. |

## Generated unit file

When `execStart` is set, 2O9 writes `/etc/systemd/system/<name>.service`:

```ini
# Generated by 2O9. Do not edit; run `209 apply` to regenerate.
[Unit]
Description=2O9-managed service: custom-app
After=network.target
Before=multi-user.target
Requires=nginx

[Service]
Type=simple
ExecStart=/nix/store/<hash>-custom-app-1.0/bin/app
Environment=PORT=8080
Environment=DB_HOST=localhost

[Install]
WantedBy=multi-user.target
```

Empty `After=`, `Before=`, and `Requires=` lines are omitted. Each
`environment` entry becomes its own `Environment=` line.

Environment values containing characters outside letters, digits, and
`-_.:/+=@` are wrapped in double quotes with `\` and `"` escaped:

```ini
Environment=GREETING="hello world"
Environment=SECRET="p@ss\"word"
```

## What `209 apply` does

2O9 sorts services by their `requires` edges with Kahn's algorithm and
processes them in that order. For each service:

1. If `execStart` is set, generate the unit file. Write it when the
   content differs from the existing file.
2. If `enable` is true, run `systemctl enable <name>`. If
   `restartOnChange` is true, the unit file changed, and the service is
   running, run `systemctl restart <name>`. If the service is not
   running, run `systemctl start <name>`.
3. If `enable` is false, run `systemctl disable <name>`.

After all per-service work, 2O9 runs `systemctl daemon-reload` once if
any unit file was written or removed.

## Removal

Services that were in the previous generation's manifest but not the
current one are torn down. For each removed service, 2O9 runs
`systemctl disable`, runs `systemctl stop` if the service is running,
and removes the generated unit file at
`/etc/systemd/system/<name>.service` if it exists. `systemctl
daemon-reload` runs once at the end.

## Errors

2O9 aborts the services step with an error in these cases:

- A service name contains characters other than letters, digits, `-`,
  `_`, `.`. Example: `services: invalid service name 'web/app': only
  letters, digits, '-', '_', '.' are allowed`.
- A `requires` entry names a service that is not declared in the
  manifest. Example: `services: service 'nginx' requires 'nonexistent'
  which is not declared`.
- The `requires` graph contains a cycle. Example: `services: service
  cycle detected: a -> b -> a`.

## Permissions

`services_apply` checks `getuid() == 0` and errors out otherwise.
`systemctl enable`, `systemctl start`, `systemctl stop`, and writes to
`/etc/systemd/system/` all need root. Run `209 apply` via `sudo`.

## Examples

### Pre-existing system service

```nix
{
  services.sshd = {
    enable = true;
    after = [ "network.target" ];
  };
}
```

2O9 runs `systemctl enable sshd` and `systemctl start sshd`. No unit
file is generated.

### Custom service with environment

```nix
{
  services."my-app" = {
    enable = true;
    requires = [ "network" ];
    type = "simple";
    execStart = "/nix/store/abc123-my-app-1.0/bin/server";
    environment = {
      PORT = "8080";
      LOG_LEVEL = "info";
    };
    restartOnChange = true;
  };
}
```

2O9 writes `/etc/systemd/system/my-app.service`, runs `systemctl enable
my-app`, and runs `systemctl start my-app`. On later `209 apply` runs,
if the unit file content changes and the service is running, 2O9 runs
`systemctl restart my-app`.

### Service with dependencies

```nix
{
  services = {
    network = { enable = true; };
    nginx = {
      enable = true;
      requires = [ "network" ];
    };
    "web-app" = {
      enable = true;
      requires = [ "nginx" ];
      execStart = "/usr/local/bin/web-app";
    };
  };
}
```

2O9 activates `network`, then `nginx`, then `web-app` in that order.

### Disabling a service

```nix
{
  services.sshd.enable = false;
}
```

2O9 runs `systemctl disable sshd`. The service is not stopped if it is
already running; use `systemctl stop sshd` to stop it manually.

## Dry run

`209 apply --dry-run` prints every `systemctl` command and every unit
file write that would happen, then exits without making changes. Use it
to preview the effect of a manifest edit before committing it.
