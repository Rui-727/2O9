# Phase 4: File Systems and Swap

2O9 manages `/etc/fstab` from declarations in `2O9.nix`. File systems and
swap devices are written to `/etc/fstab` during `209 apply`. Existing
fstab content is backed up to `/etc/fstab.bak.2O9` (single most-recent
backup) before the new file is written.

## Config schema

Declare file systems under `fileSystems`, keyed by mount point. Each
entry has `device`, `fsType`, and an optional `options` list (default
`["defaults"]`).

```nix
{
  fileSystems = {
    "/" = {
      device = "/dev/sda2";
      fsType = "ext4";
      options = [ "defaults" "noatime" ];
    };
    "/boot" = {
      device = "/dev/sda1";
      fsType = "vfat";
      options = [ "defaults" ];
    };
    "/home" = {
      device = "/dev/sda3";
      fsType = "btrfs";
      options = [ "defaults" "subvol=@" "compress=zstd" ];
    };
  };
}
```

The `device` field can be a raw block path (`/dev/sda2`), a label
(`/dev/disk/by-label/arch`), or a UUID
(`/dev/disk/by-uuid/1234-5678`). 2O9 writes it to fstab as given.

Swap devices go under `swapDevices`, a list of attrsets. Each entry has
`device` and optionally `size` (in MB). When `size` is set and `device`
is a regular file path (anything not under `/dev/`), 2O9 creates the
swap file with `dd`, sets mode 0600, and runs `mkswap` on it.

```nix
{
  swapDevices = [
    { device = "/dev/sda4"; }
    { device = "/var/swapfile"; size = 8192; }
  ];
}
```

For a swap file that already exists with the right size, 2O9 skips
creation. If the file exists with a different size, 2O9 prints a warning
and leaves the file alone.

## Generated fstab format

The fstab begins with a header comment, then the file system entries
sorted by mount point depth (shallower first), then swap entries. The
dump and fsck pass fields follow the standard convention:

- `/` gets dump=1, pass=1.
- `/boot`, `/boot/...`, `/efi`, and `/efi/...` get dump=0, pass=2.
- All other file systems get dump=0, pass=0.
- Swap entries use `none swap defaults 0 0`.

Example output:

```
# Generated by 2O9. Do not edit; run `209 apply` to regenerate.
/dev/sda2 / ext4 defaults,noatime 1 1
/dev/sda1 /boot vfat defaults 0 2
/dev/sda3 /home btrfs defaults,subvol=@,compress=zstd 0 0
/dev/sda4 none swap defaults 0 0
/var/swapfile none swap defaults 0 0
```

## Apply behavior

When `209 apply` runs and the generated content differs from the
existing `/etc/fstab`, 2O9 backs up the old file to
`/etc/fstab.bak.2O9`, writes the new fstab atomically via `mkstemp` and
`rename`, runs `swapon -a` to activate any new swap devices, and runs
`mount -a` to mount any new file systems.

`mount -a` does not remount file systems that are already mounted.
Option changes (such as adding `noatime` to an already-mounted `/`)
require a reboot to take effect. 2O9 prints a note when this applies.

## Removing a file system

When a file system declared in the previous generation's manifest is no
longer in the current manifest, 2O9 prints a warning naming the mount
point and the device it used to be on:

```
fstab: WARNING file system /home no longer declared (was mounted on /dev/sda3); fstab line removed. Unmount manually if needed.
```

2O9 does not unmount or destroy anything. The fstab line is dropped
because 2O9 only writes entries for currently-declared file systems. The
user must do any unmount or swap teardown manually.

## Permissions

2O9 writes to `/etc/fstab` and runs `mount`, `swapon`, and `mkswap`.
`fstab_apply` checks `getuid() == 0` and returns an error otherwise. Run
`209 apply` via `sudo`.

## Empty fileSystems block

If `fileSystems` is missing or empty, 2O9 generates a minimal fstab by
copying the root (`/`) entry from the existing `/etc/fstab`. If no root
entry is found, it prints a warning and writes only the swap entries (or
just the header if there are no swaps either).
