# Phase 4: PAM and NSS

2O9 writes PAM service configs and the NSS database config from declarations
in `2O9.nix`. On `209 apply`, the PAM block lands in `/etc/pam.d/<service>`
and the NSS block lands in `/etc/nsswitch.conf`. Each existing file is
backed up to `<path>.bak.2O9` (single most-recent backup) before being
overwritten.

## Config schema

PAM and NSS live under `security` in the manifest.

```nix
{
  security = {
    pam = {
      login = ''
        auth        required    pam_unix.so
        auth        optional    pam_permit.so
        account     required    pam_unix.so
        password    required    pam_unix.so sha512 shadow
        session     required    pam_unix.so
        session     optional    pam_systemd.so
      '';

      sshd = {
        auth = [
          { type = "required"; control = "pam_unix.so"; }
          { type = "optional"; control = "pam_permit.so"; }
        ];
        account = [
          { type = "required"; control = "pam_unix.so"; }
        ];
        password = [
          { type = "required"; control = "pam_unix.so"; args = [ "sha512" "shadow" ]; }
        ];
        session = [
          { type = "required"; control = "pam_unix.so"; }
          { type = "optional"; control = "pam_systemd.so"; }
        ];
      };
    };

    nss = {
      passwd = [ "files" "systemd" ];
      group  = [ "files" "systemd" ];
      shadow = [ "files" ];
      hosts  = [ "files" "myhostname" "resolve" [ "dns" ] ];
      services  = [ "files" ];
      networks  = [ "files" ];
      protocols = [ "files" ];
      rpc       = [ "files" ];
      ethers    = [ "files" ];
      netmasks  = [ "files" ];
      netgroup  = [ "files" ];
      automount = [ "files" ];
      aliases   = [ "files" ];
    };
  };
}
```

## PAM: string form

When a service's value is a string, 2O9 writes the text verbatim to
`/etc/pam.d/<service>`. The 2O9 header comment is prepended so the file is
self-identifying; the rest of the text is unchanged.

```nix
{
  security.pam.login = ''
    auth        required    pam_unix.so
    account     required    pam_unix.so
    password    required    pam_unix.so sha512 shadow
    session     required    pam_unix.so
  '';
}
```

Resulting `/etc/pam.d/login`:

```
# Generated by 2O9. Do not edit; run `209 apply` to regenerate.
auth        required    pam_unix.so
account     required    pam_unix.so
password    required    pam_unix.so sha512 shadow
session     required    pam_unix.so
```

Use the string form when you want full control over the file, when you need
directives 2O9's structured form does not cover (such as `@include` lines),
or when you are copying an existing PAM config verbatim.

## PAM: structured form

When a service's value is an attrset, 2O9 renders the file. The keys are the
four PAM facilities: `auth`, `account`, `password`, `session`. Each key
holds a list of entries. An entry has:

| Field | Required | Meaning |
|---|---|---|
| `type` | yes | One of `required`, `requisite`, `sufficient`, `optional` |
| `control` | yes | The module path, e.g. `pam_unix.so` |
| `args` | no | A list of string arguments appended after the module |

2O9 writes the facility column left-justified in a 12-character field, the
type column the same width, then the module and arguments. A facility
missing from the attrset is omitted; 2O9 does not emit an empty section.

```nix
{
  security.pam.sshd = {
    auth = [
      { type = "required"; control = "pam_unix.so"; }
      { type = "optional"; control = "pam_permit.so"; }
    ];
    account = [
      { type = "required"; control = "pam_unix.so"; }
    ];
    password = [
      { type = "required"; control = "pam_unix.so"; args = [ "sha512" "shadow" ]; }
    ];
    session = [
      { type = "required"; control = "pam_unix.so"; }
      { type = "optional"; control = "pam_systemd.so"; }
    ];
  };
}
```

Resulting `/etc/pam.d/sshd`:

```
# Generated by 2O9. Do not edit; run `209 apply` to regenerate.
auth        required    pam_unix.so
auth        optional    pam_permit.so
account     required    pam_unix.so
password    required    pam_unix.so sha512 shadow
session     required    pam_unix.so
session     optional    pam_systemd.so
```

## NSS

`security.nss` is an attrset keyed by database name (`passwd`, `group`,
`shadow`, `hosts`, `services`, `networks`, `protocols`, `rpc`, `ethers`,
`netmasks`, `netgroup`, `automount`, `aliases`). Each value is a list of
sources. A source is either a string or a sublist.

A string source becomes its own name in the line:

```nix
passwd = [ "files" "systemd" ];
```

renders as:

```
passwd: files systemd
```

A sublist source lets you attach action specifiers. The first element is
the source name. The rest are action strings (status=value). A
single-element sublist gets the default action `NOTFOUND=return`:

```nix
hosts = [ "files" "myhostname" "resolve" [ "dns" ] ];
```

renders as:

```
hosts: files myhostname resolve dns [NOTFOUND=return]
```

A sublist with explicit action strings writes them inside the brackets,
space-separated:

```nix
hosts = [ "files" [ "dns" "UNAVAIL=continue" "TRY_AGAIN=return" ] ];
```

renders as:

```
hosts: files dns [UNAVAIL=continue TRY_AGAIN=return]
```

## Apply behavior

`209 apply` writes one file per declared PAM service and one
`/etc/nsswitch.conf` from the declared NSS block. Before writing, each
existing file is copied to `<path>.bak.2O9`; the previous backup is
overwritten, so only the most recent pre-2O9 version is kept.

PAM reads `/etc/pam.d/` on every authentication call, so the new config
takes effect immediately. No reload is needed. NSS reads
`/etc/nsswitch.conf` on the next lookup by glibc; long-running daemons may
cache the previous config until they are restarted.

## Removing a service

When a PAM service appears in the previous generation's manifest but is
absent from the current one, 2O9 prints a warning naming the service and
the file path. The file is left in place; you remove it manually if you
want it gone.

```
pam_nss: WARNING PAM service 'old-service' no longer declared (was at
/etc/pam.d/old-service); leaving file in place, remove manually if desired
```

NSS has no removal concept. The current manifest fully defines
`/etc/nsswitch.conf`; databases present in the previous file but absent
from the new one simply disappear from the rewritten file.

## Skipping a block

If `security.pam` is missing or empty, 2O9 does not touch `/etc/pam.d/`. If
`security.nss` is missing or empty, 2O9 does not touch
`/etc/nsswitch.conf`. Both can be skipped in the same run.

## Permissions

`pam_nss_apply` checks `getuid() == 0` and returns an error otherwise. Run
`209 apply` via `sudo`.

## Path safety

A PAM service name containing `/` or `..` is rejected with an error and the
apply aborts. Service names map directly to filenames under `/etc/pam.d/`,
so the rejection keeps the writer inside that directory.
